

# Stripe-Ready Payment Architecture â€” Implementation Plan

## 1. Current State Analysis

### What exists today
- **`transactions` table**: flat structure with `transaction_id`, `amount`, `order_name`, `status` (paid/pending/voided/needs_review/refunded), `payment_method` (credit_card/bank_transfer/qr_promptpay/cash), FKs to members/packages/locations/staff. No VAT columns, no idempotency_key, no source_type/source_ref.
- **`transfer_slips` table**: has `amount_thb`, `status` (needs_review/approved/rejected/voided), `linked_transaction_id` FK â†’ transactions, `reviewer_staff_id`, `slip_file_url`, `bank_reference`.
- **`member_packages` table**: has `member_id`, `package_id`, `status` (text: ready_to_use/active), `sessions_remaining`, `sessions_used`, `purchase_date`, `activation_date`, `expiry_date`. No `purchase_finance_transaction_id`, no `package_name_snapshot`.
- **`member_billing` table**: `member_id`, `transaction_id` FK â†’ transactions, `amount`, `description`.
- **`activity_log` table**: `event_type`, `entity_type`, `entity_id`, `activity` (text), `old_value`/`new_value` (jsonb), `staff_id`, `member_id`.
- **Approve flow**: Client-side in `useTransferSlips.ts` â€” does 6 sequential Supabase inserts (NOT atomic/transactional). Race conditions possible.
- **No edge functions** for payments. Only `daily-briefing`, `invite-staff`, `line-auth`.
- **7 files** query `transactions` table directly (hooks, importers, settings export).

### Key problems with current architecture
1. **No atomicity**: Slip approval does 4+ separate inserts from client â€” if any fails mid-way, data is inconsistent.
2. **No idempotency**: Transaction ID generated by count â€” race condition if two approvals happen simultaneously.
3. **No VAT columns**: VAT is computed at display-time only, not stored.
4. **No Stripe readiness**: No payment_intent tracking, no webhook infrastructure, no source_type field.
5. **payment_method enum** missing `card_stripe` and `qr_promptpay_stripe`.

## 2. Design Decision â€” Additive Migration Strategy

Rather than creating entirely new tables (which would break all 7+ files reading `transactions`), we will **extend the existing tables** with new columns. This is the safest approach because:
- All existing queries continue to work unchanged
- No data migration needed
- Gradual adoption â€” new fields are nullable with defaults
- Zero regression risk on existing flows

## 3. Implementation Plan

### Phase 1: Schema Migration (additive columns)

**Migration 1 â€” Extend `transactions` table:**
```sql
-- Add Stripe/payment readiness columns
ALTER TABLE transactions ADD COLUMN IF NOT EXISTS source_type text DEFAULT 'manual';
ALTER TABLE transactions ADD COLUMN IF NOT EXISTS source_ref text;
ALTER TABLE transactions ADD COLUMN IF NOT EXISTS idempotency_key text UNIQUE;
ALTER TABLE transactions ADD COLUMN IF NOT EXISTS currency text DEFAULT 'THB';
ALTER TABLE transactions ADD COLUMN IF NOT EXISTS amount_gross numeric;
ALTER TABLE transactions ADD COLUMN IF NOT EXISTS amount_ex_vat numeric;
ALTER TABLE transactions ADD COLUMN IF NOT EXISTS amount_vat numeric;
ALTER TABLE transactions ADD COLUMN IF NOT EXISTS vat_rate numeric DEFAULT 0.07;
ALTER TABLE transactions ADD COLUMN IF NOT EXISTS discount_amount numeric DEFAULT 0;
ALTER TABLE transactions ADD COLUMN IF NOT EXISTS paid_at timestamptz;
ALTER TABLE transactions ADD COLUMN IF NOT EXISTS package_name_snapshot text;
ALTER TABLE transactions ADD COLUMN IF NOT EXISTS sold_to_name text;
ALTER TABLE transactions ADD COLUMN IF NOT EXISTS sold_to_contact text;
-- Extend payment_method enum
ALTER TYPE payment_method ADD VALUE IF NOT EXISTS 'card_stripe';
ALTER TYPE payment_method ADD VALUE IF NOT EXISTS 'qr_promptpay_stripe';
ALTER TYPE payment_method ADD VALUE IF NOT EXISTS 'other';
-- Add failed status
ALTER TYPE transaction_status ADD VALUE IF NOT EXISTS 'failed';
```

**Migration 2 â€” Extend `member_packages` table:**
```sql
ALTER TABLE member_packages ADD COLUMN IF NOT EXISTS purchase_transaction_id uuid REFERENCES transactions(id);
ALTER TABLE member_packages ADD COLUMN IF NOT EXISTS package_name_snapshot text;
ALTER TABLE member_packages ADD COLUMN IF NOT EXISTS sessions_total integer;
```

**Migration 3 â€” Add indexes:**
```sql
CREATE INDEX IF NOT EXISTS idx_transactions_source_ref ON transactions(source_ref) WHERE source_ref IS NOT NULL;
CREATE INDEX IF NOT EXISTS idx_transactions_idempotency ON transactions(idempotency_key) WHERE idempotency_key IS NOT NULL;
```

**No changes needed** to `transfer_slips` or `activity_log` â€” they already have the required structure.

### Phase 2: Server-Side Approve Slip (Edge Function)

**New edge function: `supabase/functions/approve-slip/index.ts`**

Moves the current client-side 6-step approval flow into a single server-side atomic operation:
- Validates slip status = `needs_review`
- Generates idempotency_key from `slip:{slipId}`
- Creates transaction with all new fields (VAT calculation, snapshot, source_type='transfer_slip', source_ref=slipId)
- Creates member_billing
- Creates member_package with `purchase_transaction_id` link
- Updates slip status to `approved` with `linked_transaction_id`
- Writes activity_log entry
- All wrapped in a DB function or sequential with idempotency guard
- Returns the created transaction

**Config:** `verify_jwt = false` (validate in code with `getClaims()`)

### Phase 3: Stripe Checkout Edge Function (Stub)

**New edge function: `supabase/functions/stripe-create-checkout/index.ts`**

- Validates auth via `getClaims()`
- Accepts: `member_id`, `package_id`, `location_id`
- Looks up package price
- Creates a `transactions` row with `status: 'pending'`, `source_type: 'stripe'`, `idempotency_key`
- Calls Stripe API to create Checkout Session (stubbed until Stripe secret is added)
- Stores `source_ref = checkout_session_id`
- Returns `{ checkout_url }`

### Phase 4: Stripe Webhook Edge Function (Stub)

**New edge function: `supabase/functions/stripe-webhook/index.ts`**

- `verify_jwt = false` (public endpoint, verified by Stripe signature)
- Handles `checkout.session.completed`:
  - Finds transaction by `source_ref`
  - Updates status â†’ `paid`, sets `paid_at`
  - Creates `member_package` entitlement
  - Creates `member_billing` record
  - Logs to `activity_log`
- Handles `charge.refunded`:
  - Finds transaction, marks `refunded`
  - Optionally deactivates member_package
- Idempotent via `idempotency_key` check

**Stripe secret**: Will use `secrets--add_secret` to request `STRIPE_SECRET_KEY` and `STRIPE_WEBHOOK_SECRET` when user is ready to connect.

### Phase 5: Refactor Client Hooks (Minimal)

**`src/hooks/useTransferSlips.ts`** â€” `useApproveSlip`:
- Replace the 6-step client-side flow with a single call: `supabase.functions.invoke('approve-slip', { body: { slipId, packageId, note } })`
- Keep all existing query invalidation and toast logic
- No changes to reject/void (they're simple single-table updates, low risk)

**No changes to**:
- `useFinanceTransactions` â€” existing `select('*')` will automatically include new columns
- `useTransferSlipsList` â€” no change needed
- `useMemberBilling`, `useMemberPackages` â€” reads still work

- Import/export files â€” new columns are nullable, exports continue working
