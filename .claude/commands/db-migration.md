You are authoring a Supabase migration for **$ARGUMENTS**.

> **Preferred flow:** schema changes go through Lovable AI or Supabase Studio — Lovable then auto-writes the migration file for you. Use this command ONLY when hand-authoring is necessary (complex multi-statement migration, policy fix, data backfill, etc.). Confirm with the user this is the right path before proceeding.

## Step 1 — Confirm the change set

Ask (one `AskUserQuestion`):
1. **Migration type** — new table / alter table / new policy / data backfill / function-or-trigger?
2. **Affected tables** — list them
3. **Access model** — which `access_level` can read/write? Is it member-owned data (→ `get_my_member_id()`) or staff data (→ `has_min_access_level()`)?
4. **Realtime** — should it stream to the frontend? (if yes, plan to add the table to `TABLE_INVALIDATION_MAP` in `src/hooks/useRealtimeSync.ts` in a follow-up edit)

## Step 2 — Pick the filename

Use the exact Lovable convention:

```
supabase/migrations/<YYYYMMDDHHMMSS>_<uuid-v4>.sql
```

- Timestamp = UTC now, format `date -u +%Y%m%d%H%M%S`
- UUID v4 = `uuidgen | tr '[:upper:]' '[:lower:]'`
- Read the most recent file in `supabase/migrations/` to mirror the exact naming shape.

## Step 3 — Draft the SQL (show to user BEFORE writing the file)

Use this skeleton. Fill in concrete names. Never include `DROP` or `ALTER ... DISABLE ROW LEVEL SECURITY`.

```sql
-- <one-line description of the change>

-- 1. Schema change (CREATE TABLE / ALTER TABLE / CREATE INDEX / etc.)
create table public.<snake_case_name> (
  id uuid primary key default gen_random_uuid(),
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now(),
  -- domain columns here (snake_case)
);

-- 2. RLS — MANDATORY for every new table
alter table public.<snake_case_name> enable row level security;

-- 3. Policies — at least one per operation the frontend needs
create policy "<name> readable by level_2+"
  on public.<snake_case_name>
  for select
  using (public.has_min_access_level('level_2_operator'));

create policy "<name> writable by level_3+"
  on public.<snake_case_name>
  for insert with check (public.has_min_access_level('level_3_manager'));

-- For member-owned data, use get_my_member_id() instead:
-- using (member_id = public.get_my_member_id())

-- 4. Triggers (updated_at, activity log, etc.) as needed
create trigger set_<name>_updated_at
  before update on public.<snake_case_name>
  for each row execute function public.tg_set_updated_at();
```

Show this draft to the user and get approval before writing the file.

## Step 4 — Write the file

- Write to the timestamped path from Step 2.
- Do NOT edit any migration that is already deployed to prod (append-only). If a rollback is needed, write a NEW reverse migration.
- Do NOT modify `supabase/config.toml` project-level settings. Per-function blocks may be added if registering a new edge function.

## Step 5 — Post-migration wiring

- If the table needs realtime → edit `src/hooks/useRealtimeSync.ts` `TABLE_INVALIDATION_MAP` and list the query-key prefixes to invalidate.
- If the table needs a frontend hook → add query keys to `src/lib/queryKeys.ts` and a hook in `src/hooks/use*.ts`.
- Types in `src/integrations/supabase/types.ts` are auto-regenerated by Lovable — do NOT hand-edit.

## Step 6 — Report

List: migration filename, tables touched, policies added (names + rule summary), follow-up edits (`TABLE_INVALIDATION_MAP`, `queryKeys.ts`, hook).
